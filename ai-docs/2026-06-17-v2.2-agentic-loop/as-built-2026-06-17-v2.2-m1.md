## 1. What we set out to do

Make the v2.0 `agents/` A2A mesh **start its own work once a day, unattended, on Cloudflare** — pick a topic, run generate → migrate → evaluate **to a preview and stop** (never auto-publish), and surface the result as a draft any human can see. The "agent decides to begin" front door the mesh never had.

Two hard constraints shaped the whole design:
1. **Everything sleeps.** On Cloudflare the containers scale to zero (`sleepAfter` 15–30m). From cold, the first hit **errors** rather than waits — in practice Jack manually refreshes until each wakes. An unattended loop has nobody to refresh.
2. **15-minute ceiling on internal timers.** Cron Triggers, DO alarms, and Queue consumers are hard-capped at 15 min wall time — but a real Kimi migration runs 7–35 min. So the naïve "add a cron that holds the run open" fails twice (no keepalive + cut at 15 min).

## 2. The decisions (locked 2026-06-17)

| Choice | Decision | Why |
|---|---|---|
| **Trigger** | **O5 — GitHub Actions cron** (not the on-platform O4 Workflow) | Decoupled + first-class observability: the GH run log *is* the audit trail. O4 Workflow stays as the later durability upgrade. |
| **Keepalive** | **Fire (`blocking:false`) + poll** `/store/runs/:id` every 45s | Each poll is an incoming request that renews the coordinator's `sleepAfter` — the poll *is* the heartbeat; robust to a dropped connection across a 30-min run. |
| **Identity** | **NULL `user_email` system run** (mesh token, no SSO, no `requestedBy`) | The mesh already treats `user_email IS NULL` as a shared/public lane visible to every signed-in user. No fake account, no schema change. |
| **Output** | **Preview-only** (`.aem.page`), never `.aem.live` | Verified: the deployed MCP `preview_publish` only POSTs to `admin.hlx.page/preview/...`; there is **no `/live/` call anywhere in the mesh**. So "don't auto-publish" is satisfied *by construction* — no mesh change needed. |
| **Topic** | **Agent-led ideation** (coordinator asks content-gen when no topic) | The mesh starts its own work; deterministic/date-seeded so the scheduled job can't flake. |
| **Backend / site / route** | Real Kimi (opencode) · `jackzhaojin/da-live-postal-2025-07` · `full-loop` (attaches an eval score to each draft) · fanOut 1 | |

## 3. What we built (shipped this session)

**Agent-led ideation (the self-starting front door)**
- `content.ideate` skill in **content-gen** — `generator.ts` `ideateTopic()` + wired in `index.ts`. Deterministic, date-seeded, lane-aware (default lane `postal-logistics`), template tier (no creds, no network in the critical path). Contract `agents/contracts/content.ideate.v1.json`.
- **Coordinator** now ideates when a generate route arrives with **no `topic`** (`coordinator/src/executor.ts`: `willIdeate`, `validateForRoute({skipTopic})`) — it asks content-gen `content.ideate`, writes the chosen topic back into the run config, and proceeds. Contract `coordinate.run.v1.json` updated (topic now optional).

**The GitHub Actions daily loop (O5)**
- `.github/workflows/daily-content-loop.yml` — `schedule` (daily **09:20 UTC**) + `workflow_dispatch` (overrides; `dryrun` backend for cheap smokes).
- `.github/scripts/daily-content-loop.mjs` — dependency-free Node driver: **pre-warm** (curl `/health` ×4 with retry/backoff — public, woke the cold cascade for the operator) → **submit** `coordinate.run` (Bearer `A2A_MESH_TOKEN`, `blocking:false`, **no `requestedBy`**, **no `topic`**, config `source:daily-loop`) → **resolve** runId via `/store/runs?contextId=` → **poll-keepalive** `/store/runs/:id` to terminal → **step summary** (topic + preview URL + score). Tokens live in GH repo secrets.
- **Self-heal:** auto-retries once (`MAX_ATTEMPTS=2`) so a cold-migration first turn or an occasional Kimi stall recovers on now-warm containers; a genuine failure still reports red (`completed_with_failures`).

## 4. End-to-end validation on Cloudflare

`workflow_dispatch` runs the *identical* job that `schedule` fires, so dispatch is a faithful proof of the scheduled path.

- ✅ **Wake-up** — pre-warm woke all 4 cold containers every run (healthy on attempt 1 within seconds).
- ✅ **Scheduled job (dryrun)** — full loop in **92s**; agent-led ideation picked *"A practical guide to postal address validation and standardization"*; generate → migrate → eval, score **75**, preview URL, shared system run.
- ✅ **Real loop (opencode/Kimi)** — run `20a3cf2a` in **454s**: **Kimi K2.6 authored a real da.live page to `.aem.page` preview** (never `.aem.live`), eval **73**, `user_email` NULL (shared/visible to all). GH step summary surfaced topic + preview URL + score.

## 5. The one known reliability risk

**Cold migration container.** Because everything sleeps between daily runs, *every* daily run hits a cold migration container, and the **first** opencode turn on a fresh-cold container can exceed the 20-min `OPENCODE_MIGRATION_TIMEOUT_MS` (observed: first post-deploy run timed out; the warm retry succeeded in ~7.6 min). **Mitigation shipped:** the loop auto-retries once on now-warm containers — what recovers it in practice. **Deeper future fix:** a migration-agent `/warmup` endpoint to boot `opencode serve` + MCP during pre-warm so the first real turn isn't paying cold-init.
